# A10:2025 Mishandling of Exceptional Conditions — deep

Pull when an error-handling finding is contested or applicability is unclear. New in 2025;
also carries part of the former SSRF category.

**Check**
- **Fail closed:** on error/timeout/resource exhaustion the system denies, it does not fall
  open to an allow/authenticated state.
- External error responses are generic; details (stack traces, internal IDs, secrets) are
  logged internally, never returned to the client.
- Exceptional paths (null, empty, oversized, malformed, downstream-down) are handled, not crash.
- Resource limits/timeouts/circuit-breakers on outbound calls; no unbounded work from one request.

**Pitfalls**
- A `catch` that logs and then continues as if authorised (fail-open).
- Returning the raw exception message to the user (info disclosure).
- A NULL-dereference or unhandled error that takes the service down (DoS).

**Example / anti-example**
- *Bad:* auth service times out, the gateway treats the call as "allow".
- *Good:* timeout → deny + generic 503 to client, detail logged, circuit-breaker trips.
