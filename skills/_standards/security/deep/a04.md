# A04:2025 Cryptographic Failures — deep

Pull when a crypto finding is contested or applicability is unclear.

**Check**
- Sensitive data (PII, secrets, tokens) encrypted in transit (TLS 1.2+ enforced) and at rest.
- Strong, current algorithms (AES-GCM, SHA-256+); no MD5/SHA-1/DES/ECB; no home-rolled crypto.
- Keys managed in a KMS/HSM/secure enclave; rotated; never in source or config files.
- Passwords hashed with a slow, salted KDF (argon2/bcrypt/scrypt), never encrypted or plain.
- Randomness from a CSPRNG for tokens/IDs/IVs.

**Pitfalls**
- TLS terminated at the edge but plaintext on the internal hop.
- Secrets committed to the repo or baked into a container image.
- Reusing a static IV/nonce with a stream/GCM cipher.

**Example / anti-example**
- *Bad:* user PII stored unencrypted "because the DB is private".
- *Good:* field-level encryption with KMS-managed keys + TLS everywhere, defence in depth.
