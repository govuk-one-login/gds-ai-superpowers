# A09:2025 Security Logging and Alerting Failures — deep

Pull when a logging/alerting finding is contested or applicability is unclear. (Renamed to
emphasise *alerting*, not just logging.)

**Check**
- Security-relevant events logged: auth success/failure, access-control denials, input-validation
  failures, high-value transactions — with enough context to investigate.
- **No PII or secrets in logs** (filter tokens, passwords, personal data before shipping).
- Logs are tamper-resistant, time-synced, and centralised; retention meets policy.
- **Alerting** on suspicious patterns wired to a responder — detection without alerting is a gap.
- Aligns with NCSC CAF Security Monitoring.

**Pitfalls**
- Logging everything including PII/tokens (turning a log store into a breach target).
- Logging but never alerting — nobody sees the signal.
- No correlation id, so an incident can't be traced across services.

**Example / anti-example**
- *Bad:* failed logins not recorded; full request bodies (with passwords) dumped to logs.
- *Good:* structured auth events with a correlation id, PII filtered, alert on brute-force.
