# A03:2025 Software Supply Chain Failures — deep

Pull when a supply-chain finding is contested or applicability is unclear. (Broader successor
to the old "Vulnerable and Outdated Components".)

**Check**
- Dependencies pinned/locked; an SBOM or lockfile exists; provenance is known.
- Automated vulnerability scanning of deps in CI, gating on severity.
- Build pipeline integrity: signed/verified artifacts, least-privilege CI tokens, no
  unreviewed third-party CI actions with broad scope.
- Third-party scripts/registries vetted; typosquat/dependency-confusion risk considered.

**Pitfalls**
- Treating this as "just patch old libraries" — it covers the whole chain (build, CI, registries).
- A CI action pinned to a mutable tag (`@v3`) rather than a commit SHA.
- Internal package names resolvable from a public registry (dependency confusion).

**Example / anti-example**
- *Bad:* CI uses `some/action@main`, granting it repo-write secrets.
- *Good:* the action is pinned to a SHA, runs with a scoped token, and deps are scanned + SBOM'd.
