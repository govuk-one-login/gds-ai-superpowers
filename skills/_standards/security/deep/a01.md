# A01:2025 Broken Access Control — deep

Operational depth for the lens line. Pull when an access-control finding is contested or its
applicability is unclear.

**Check**
- **Deny by default.** Every resource/route requires a positive authorisation decision; the
  default for an unmatched rule is deny, not allow.
- **Server-side, per-object.** Authorisation is enforced on the server for *every* object
  access (not just the list/landing route), checking the authenticated principal against the
  specific object's ownership/tenant — not a role claim sent by the client.
- **No IDOR.** Direct object references (`/orders/{id}`) are validated against the caller's
  entitlement, not just authenticated-and-exists.
- **SSRF egress (now absorbed here).** Server-side fetches of user-supplied URLs are
  allow-listed and blocked from internal/metadata ranges (169.254.169.254, RFC1918).

**Pitfalls**
- Authorising the collection endpoint but not the item endpoint behind it.
- Trusting a `role`/`isAdmin` field from the token body or a header the client controls.
- Predictable/ sequential IDs that make IDOR trivial to enumerate.
- Path traversal (`../`) reaching files outside the intended root.

**Example / anti-example**
- *Bad:* `GET /api/orders/{id}` returns the order whenever `{id}` exists and the caller is
  logged in. Any user reads any order (IDOR).
- *Good:* the query is scoped — `WHERE id = :id AND owner_id = :caller` — so a mismatched
  owner returns 404, not the record.
