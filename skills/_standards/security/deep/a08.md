# A08:2025 Software or Data Integrity Failures — deep

Pull when an integrity finding is contested or applicability is unclear.

**Check**
- Updates/artifacts/plugins are signed and signatures verified before use (no unsigned auto-update).
- No insecure deserialization of untrusted data (no native deserialize of attacker-controlled bytes).
- CI/CD integrity: the deployed artifact is the reviewed one; provenance/attestation where possible.
- Critical state changes are tamper-evident (signed, logged, integrity-checked).

**Pitfalls**
- Deserializing JSON/XML/native objects into types that trigger code paths on construction.
- Trusting an unsigned update channel or a CDN script with no SRI hash.
- Treating this as the same as A03 — here the focus is *verification of integrity*, not the source.

**Example / anti-example**
- *Bad:* a third-party `<script>` with no `integrity=` SRI attribute.
- *Good:* SRI-pinned scripts; signed releases verified on deploy; no native deserialize of input.
