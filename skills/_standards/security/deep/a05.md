# A05:2025 Injection — deep

Pull when an injection finding is contested or applicability is unclear. (Includes XSS.)

**Check**
- Untrusted input never concatenated into an interpreter context (SQL, NoSQL, OS command,
  LDAP, ORM raw, template). Use parameterised queries / prepared statements / safe APIs.
- Output encoding contextual to the sink (HTML, attribute, JS, URL) to stop XSS; a CSP as
  defence in depth.
- Allow-list validation of structure/type/range at the boundary (not a deny-list of "bad chars").

**Pitfalls**
- Parameterising most queries but building one dynamic `ORDER BY`/table name by string concat.
- Relying on client-side validation only.
- Escaping for HTML body but injecting into a JS or attribute context.

**Example / anti-example**
- *Bad:* `"SELECT * FROM u WHERE name='" + name + "'"`.
- *Good:* `SELECT * FROM u WHERE name = ?` with a bound parameter.
