# A06:2025 Insecure Design — deep

Pull when an insecure-design finding is contested or applicability is unclear. This is an
*architecture/logic* flaw, not an implementation bug — its presence in the Top 10 is itself
the argument for threat-modelling at design time.

**Check**
- The design has threat-modelled its abuse cases (not just happy paths) — `threat-model` ran.
- Security controls are designed into flows that need them: rate limits, anti-automation,
  segregation of duties, secure password-reset/account-recovery, business-logic limits.
- Trust boundaries are explicit; no flow assumes an upstream did a check it didn't.

**Pitfalls**
- A password-reset flow that reveals whether an account exists, or with a guessable token.
- No limit on a costly operation (e.g. unbounded refund, unthrottled OTP send).
- Treating this as fixable by better coding — it needs a design change.

**Example / anti-example**
- *Bad:* "transfer funds" trusts a client-supplied `fromAccount` without re-checking ownership.
- *Good:* the server derives the source account from the session; limits + step-up auth on value.
